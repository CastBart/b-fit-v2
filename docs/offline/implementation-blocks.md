# B-Fit Offline-First PWA — Implementation Blocks

This document splits the full offline-first PWA plan into discrete, independently verifiable implementation blocks. Each block has its own scope, file list, verification steps, and exit criteria. **Do not start a block without explicit confirmation** — the process is designed so every block is a stop-and-review checkpoint.

The source of the architectural decisions is the full plan at `C:\Users\Bartosz\.claude\plans\purrfect-twirling-hammock.md`. This document is the execution view of that plan.

## Block Order and Dependencies

```
Block 1: PWA Foundation            ─┐
                                     ├─► Block 3: Critical-Data Prefetch
Block 2: Persisted Cache + Reads   ─┤
                                     ├─► Block 5: Offline Session Completion
Block 4: Service Layer + Routes    ─┤
                                     └─► Block 6: Offline Exercise CRUD
                                                 ↓
                                         Block 7: Verification Sweep
```

- Blocks 1, 2, and 4 are independent and could in principle be reordered, but the listed order is the safest path: the SW and manifest first, then the persisted cache (which renders visible value immediately), then the shared service layer that underpins offline writes.
- Block 3 depends on Block 2 (no point prefetching into an unpersisted cache).
- Blocks 5 and 6 depend on both Block 2 (provider) and Block 4 (routes).
- Block 7 is a final verification pass over everything.

#### Blockers encountered and how they were resolved

1. **`tsconfig.json` `types` field is a whitelist, not an allowlist.**
   Initial attempt followed the Serwist docs literally and added `"types": ["@serwist/next/typings"]` to `compilerOptions`. This broke typecheck across the entire project because setting `types` at all disables the default auto-discovery of every `@types/*` package (`@types/node`, `@types/react`, etc. all vanished).
   **Fix:** Drop the `types` field entirely. Keep only `"webworker"` in `lib`. The SW source (`src/app/sw.ts`) still typechecks because it declares `ServiceWorkerGlobalScope` locally and imports Serwist types directly from the package.
   **Lesson for future blocks:** Never blindly add `compilerOptions.types` — if a library's docs ask for a types entry, either put it in a localized `tsconfig` override for just the SW file, or rely on direct imports instead. This is a repo-wide footgun.

2. **Serwist + Turbopack incompatibility in Next.js 16.**
   Next 16 defaults to Turbopack. The first production build errored out with:

   > `This build is using Turbopack, with a webpack config and no turbopack config.`
   > Serwist injects a webpack plugin, which Turbopack won't run.
   > **Fix:** Changed the `build` script to `next build --webpack`. Dev (`next dev`) is unaffected because `withSerwist` is `disable`d in development, so Turbopack still handles HMR.
   > **Lesson:** Every Next 16 PWA setup will hit this until Serwist ships a Turbopack adapter. Document it loudly in the README before any deploy. Bundle analysis and build times will feel different in CI vs dev from now on.

3. **Pre-existing Next 16 PageProps violation in `workouts/builder/page.tsx`.**
   The build failed with:

   > `Type 'OmitWithTag<WorkoutBuilderPageProps, keyof PageProps, "default">' does not satisfy the constraint '{ [x: string]: never; }'. Property 'editWorkoutId' is incompatible with index signature.`
   > Root cause: `workouts/builder/page.tsx` was serving double duty — it was both the route component for `/workouts/builder` _and_ a default-exported component imported by two other files (`workouts/builder/[id]/page.tsx` passing `editWorkoutId`, and `clients/[id]/workouts/create/page.tsx` passing `forClientId`). Next 16 enforces that a `page.tsx` default export must match `PageProps` exactly, with no extra props allowed.
   > This is a **pre-existing bug** unrelated to Block 1 but blocking the build.
   > **Fix:** Extracted the implementation into `src/app/(dashboard)/workouts/builder/WorkoutBuilder.tsx` (colocated non-route file — Next only routes `page`/`layout`/`route`, other files are free). Rewrote `workouts/builder/page.tsx` to be a thin `<WorkoutBuilder />` wrapper. Updated the two importers:
   - `workouts/builder/[id]/page.tsx` → `import WorkoutBuilder from '../WorkoutBuilder'`
   - `clients/[id]/workouts/create/page.tsx` → `import WorkoutBuilder from '@/app/(dashboard)/workouts/builder/WorkoutBuilder'`
     **Lesson:** Next 16 migration is going to surface more of these. Any `page.tsx` that accepts custom props (rather than only `{ params, searchParams }`) is now invalid. Grep for `default function Page` files that take unusual prop shapes before the next block lands. Also — colocating non-route files next to `page.tsx` is the cleanest escape hatch; don't reach for a new `components/features/**` path just to unblock a type error.

4. **Middleware deprecation warning.**
   Next 16 prints: `The 'middleware' file convention is deprecated. Please use 'proxy' instead.`
   Not a blocker for Block 1. Tracked as a Next-16-housekeeping follow-up — rename `src/middleware.ts` to `src/proxy.ts` (or whatever the new convention is) in a dedicated migration commit, not mixed in with offline work.

#### Blockers encountered and how they were resolved

1. **`shouldDehydrateMutation: mutation.state.status === 'paused'` is a type error.**
   React Query's `MutationStatus` type is `'idle' | 'pending' | 'success' | 'error'` — there is no `'paused'` status. Paused-ness is tracked separately on `mutation.state.isPaused: boolean`. The comparison failed typecheck with:

   > `This comparison appears to be unintentional because the types 'MutationStatus' and '"paused"' have no overlap.`
   > **Fix:** Use `mutation.state.isPaused` instead. Semantically equivalent, typechecks cleanly.
   > **Lesson:** Paused and status are orthogonal in React Query 5. A mutation can be `pending + isPaused: true` (the network-mode-offline state) or `pending + isPaused: false` (actively in flight). Don't ever check status for paused-ness; always use `isPaused`.

2. **`idb-keyval` `get` returns `undefined`, persister expects `null`.**
   Wiring `storage: { getItem: get, setItem: set, removeItem: del }` naively (as some docs suggest) produces a subtle bug: the persister treats `undefined` as "storage not ready" and retries, while `null` means "nothing stored, use cold cache". Fresh installs would silently skip rehydration.
   **Fix:** Inline adapter that coerces: `getItem: async (k) => (await get<string>(k)) ?? null`. Same pattern on `setItem` / `removeItem` to ensure void returns.
   **Lesson:** Any storage adapter for the persister needs explicit `undefined → null` coercion. Add a comment so the next editor doesn't "simplify" this back to the broken form.

3. **`gcTime` on per-hook overrides was shorter than `maxAge`.**
   `useSession` had `gcTime: 10m` and `useSessions` had `gcTime: 5m`. These override the queryClient default. When the persister rehydrates a cache entry from yesterday, React Query immediately garbage-collects it because the per-hook `gcTime` has long expired — so offline reads would work for the first ~10 minutes after persistence, then silently fail.
   **Fix:** Bumped both to 24 days, matching the persister `maxAge`. Any other persisted hook with a non-default `gcTime` needs the same treatment — grep for `gcTime:` in `src/hooks/queries/` before adding new hooks.
   **Lesson:** Per-hook `gcTime` is a trap in a persisted cache. The rule is: persisted hooks must set `gcTime >= persister.maxAge`, or leave it unset so the provider default (24d) applies.

4. **Non-persisted hooks that weren't in the plan's exclusion list.**
   Grep surfaced three hooks not listed in `NON_PERSISTED_QUERY_KEYS`: `useClientDetail`, `useMyPT`, `useUserProfile`. All three hold account-scoped data that could leak across accounts if persisted to IDB and then read after a logout. Added them to the exclude set.
   **Lesson:** Default-deny would be safer than default-allow. The current filter persists anything whose first key segment matches the allowlist — a new hook using a new key root is off by default, which is correct. But any hook reusing an existing persisted root is on by default and must be audited by hand.

The mutation defaults run `onMutate` per execute call, so the optimistic insert would duplicate — but the eventual `setQueriesData` replacement + `onSettled` invalidation erase the duplicate as soon as one of them confirms. In practice the user cannot double-tap because the button is `disabled={isCommitting}` inside the same tick.
6. **The completion mutation's `onError` while offline is silent.** A paused mutation does not surface errors (it hasn't tried yet). When it resumes and genuinely fails, `onError` fires with the user now online — the hook's `if (!onlineManager.isOnline()) return` guard is a paranoid double-check against edge cases where the error fires during a brief reconnect flicker.

- `src/lib/pwa/mutation-defaults.ts` — three new registrations:

**Blockers / subtleties**

1. **Cannot dispatch Redux from a module.** `src/store/store.ts` exports a `makeStore()` factory and `ReduxProvider.tsx` instantiates per-request. The rewriter module has no access to a live store, so the plan's `store.dispatch(rewriteExerciseRef(...))` pattern is unworkable. Solved by the emitter bridge: the rewriter fires an event, and a React hook inside the PWA bootstrap subscribes and dispatches. Same pattern was used in Block 5 for `clearSessionBackup`.
2. **`mutateAsync` cannot be used offline.** `mutateAsync` waits for the `mutationFn` to settle; offline mutations pause and never settle until reconnect. The drawer must close immediately. Solution: allocate the tempId **in the caller**, fire-and-forget via `.mutate()`, and synchronously hand back a locally-built optimistic `Exercise` shape so downstream consumers have something to work with. The `_pending` flag is not added to the caller's optimistic row (only to the cache copy) because callers like the workout builder care about `{ id, name }` and the pending flag would trip its type.
3. **`ExerciseListResponse.exercises`, not `.items`.** The master plan pseudocode referenced `old.items`, but the actual shape returned by `getExercises` is `{ exercises: ExerciseEntity[], total, page, limit, totalPages }`. Both the `onMutate` optimistic patches and the rewriter operate on `.exercises`. Getting this wrong produces silent cache no-ops.
4. **`['exercises', key]` is paginated, not infinite.** No `pages[]` structure to walk — plain `setQueriesData` with a prefix match is sufficient. All filter variants get patched together, which is simpler and safer than trying to scope by filter.
5. **`idMap.waitFor` semantics.** If `waitFor` is called for a tempId that will never get a real id (e.g., the user deleted the optimistic exercise before the create could sync), the promise will hang forever. In practice this is fine because the mutation resumption path is itself bounded by the session: if the user closes the tab, the paused mutation persists to IDB and the next boot either resolves the waiter after the create succeeds or the whole thing is garbage-collected by `buster: APP_VERSION` on deploy. If this becomes a real issue, a future pass can add a 30s timeout + reject.
6. **The mutation defaults shape for create variables is `{ input, tempId }`.** This intentionally diverges from the standard single-arg mutation pattern because we need the caller to own tempId generation (so the optimistic entity is addressable before the mutation runs). Callers that forget to pass `tempId` will fail at runtime with a clear error from the mutationFn deref.
7. **Detail cache priming.** `onMutate` seeds `['exercise', tempId]` with the optimistic row so `useExercise(tempId)` resolves offline. On rewrite, `removeQueries(['exercise', tempId])` tears it down and `setQueryData(['exercise', real.id], real)` re-seeds under the real key. This is critical for any UI that routes to an exercise detail page immediately after offline creation.

### Blockers encountered and how they were resolved

1. **Vary-header mismatch swallowed every offline navigation.** First implementation cached the seven routes correctly into `pages-rsc`, yet offline clicks on warmed routes still rendered `/~offline`. Confirmed via DevTools that entries existed but `cache.match()` returned `undefined` because Next.js RSC responses include `Vary: RSC, Next-Router-State-Tree, Next-Router-Prefetch, Next-Url`. The warming fetch only sets `RSC: '1'`; real-nav requests carry a route-specific `Next-Router-State-Tree` value the warming fetch cannot reproduce. Cache API's default Vary-aware match failed, NetworkFirst returned nothing while offline, and the document fallback fired.
   **Fix:** `matchOptions: { ignoreVary: true }` on both RSC `NetworkFirst` handlers in `sw.ts`. Scope deliberately limited to RSC — leaves the default cache and API handlers on standard HTTP semantics.
   **Lesson:** When a `Vary` header is present and the client cannot reproduce all of its discriminator values, Cache Storage's default match is silent-failure-prone. Inspect `Vary` early when offline reads from a populated cache mysteriously miss.

2. **First-install service-worker race.** With the Vary fix in place, only `/dashboard` was cached on a clean-install login; the other six routes appeared only after a manual reload. Root cause: `useRegisterSW` and `usePrefetchCriticalData` mount in parallel from `PWAClientBootstrap`, so the warming fetches dispatched before the SW had activated and claimed the client. Fetches issued from an uncontrolled document bypass the SW entirely and never reach the cache. `/dashboard` was cached because Next.js's own router/`<Link>` prefetch traffic continued for the current route after the SW eventually claimed.
   **Fix:** `waitForServiceWorkerControl()` helper. Awaits `navigator.serviceWorker.ready` (5 s cap) and then verifies `navigator.serviceWorker.controller` is set; if not, listens for one `controllerchange` event (2 s cap). Returns false on timeout / unsupported environments, in which case warming is skipped — RSC cache wouldn't have entries anyway without an active SW.
   **Lesson:** `serviceWorker.ready` resolves on activation but does not wait for `clients.claim()` to complete. When `clientsClaim: true` is the activation strategy, `controllerchange` is the correct signal that the document is now controlled.

3. **Soft-nav login left `useSession()` stuck on `unauthenticated`.** With both fixes above, the warming pipeline still didn't fire after a clean-install login until the user manually refreshed. Root cause: `LoginForm` uses `router.push(callbackUrl); router.refresh()` (soft navigation, no remount). The auth cookie is set on the server response, but next-auth's `SessionProvider` is configured with `refetchOnWindowFocus={false}`, `refetchWhenOffline={false}`, no `refetchInterval`, and consumes the `session` prop only as `useState` initial value — re-rendering with a different prop does not propagate. So `useSession()` kept returning `{ status: 'unauthenticated' }`, the sync effect's `status !== 'authenticated'` gate stayed shut, and warming never ran.
   **Fix:** Inside `usePrefetchCriticalData`, a second one-shot effect now calls `useSession().update()` exactly once when `status === 'unauthenticated'` AND the current pathname is not in the public-route prefix list. That forces `SessionProvider` to refetch `/api/auth/session`, which reads the cookie, returns the real session, flips internal state, and propagates `status === 'authenticated'`. The existing `[queryClient, status]` dep array on the sync effect then re-runs on its own — no other code path needed. Guarded by `hasNudgedSessionRef` so it cannot loop.
   **Lesson:** Next-auth's client-side `useSession()` is not a live mirror of the cookie. Soft-nav-only login flows must explicitly trigger `update()` somewhere or `SessionProvider` will lag indefinitely. The path-aware nudge inside the prefetch hook keeps the fix self-contained without touching `LoginForm` or `SessionProvider`.

### Audit findings (preserved here so future PRs don't re-discover)

- **No `clientId` on `Workout`/`WorkoutExercise` before this PR.** Block 6 added it for `Exercise`; PR2 extended the pattern. Migration is additive and safe (nullable + unique).
- **`Workout` has no relation to plans in the schema.** PR1.5 already noted this; restated for PR2 because the original plan suggested patching `['activePlanDashboard']` on workout-update. There is nothing to patch — `PlanDay` embeds `PlanDayExercise` records directly, no workout reference.
- **`Workout.createdById` is non-nullable.** Optimistic create rows must populate `createdBy.id` correctly so `WorkoutBuilder`'s client-context detection (`existingWorkout.createdBy.id !== authSession.user.id`) doesn't false-positive route to a client tab. PR2 passes `userId` through the create variables and the optimistic builder writes it.
- **`@@unique([workoutId, order])` on `WorkoutExercise`.** Pre-existing constraint. Optimistic reorder + sync-exercises must produce a contiguous `0..N-1` order set. Existing `syncWorkoutExercises` does updates in `Promise.all` which can violate the unique constraint mid-transaction; PR2 preserves this behaviour but flags it as a latent risk if reorders ever fail in production.
- **List-shape vs detail-shape divergence.** `['workouts', 'all']` carries minimal exercises (`{id, exercise: {primaryMuscleGroup, secondaryMuscleGroups}}` + `exerciseCount`). `['workout', id]` carries the full nested exercises with parameters. Optimistic builders must produce both shapes correctly.
- **Server actions vs offline routes.** Existing online `createWorkout`/`updateWorkout`/`deleteWorkout`/`syncWorkoutExercises` server actions in `src/server/actions/workouts.ts` were left unchanged — they keep working for any non-offline-aware UI path. Duplication with the new `workoutService` is intentional for PR2 scope. A later cleanup can delegate the actions to the service if both stay in lockstep.

### Blockers encountered and how they were resolved

1. **`idMap.waitFor` had no rejection path.** A permanently-failing exercise create (e.g. 4xx validation error) would leave dependent mutations (workout-create with that exercise as a `tmp_*` reference, exercise-update/delete queued behind the failed create) hanging forever on `idMap.waitFor`. With React Query's `networkMode: 'online'`, `onError` only fires after retries are exhausted on a non-paused mutation, so the rejection signal is reliable when it does fire.
   **Fix:** added `idMap.reject(tempId, reason)`. Rejects all current waiters for that tempId; does not write a rejection marker to the persisted mirror, so a later successful retry of the create still resolves new waiters. Wired into `['exercises','create']` and `['workouts','create']` `onError`.
   **Lesson:** any `waitFor`-style primitive needs an explicit failure path. Silent hangs are worse than visible errors.

2. **Online-only gating inside `mutationFn` is dead code when offline.** With `networkMode: 'online'` (the project default), an offline `mutate()` enters paused state BEFORE `mutationFn` runs. A `requireOnline()` check inside `mutationFn` never fires offline → no toast, but the paused mutation is still persisted to IDB (`shouldDehydrateMutation` persists every paused mutation, regardless of `mutationKey`) and silently replays on reconnect.
   **Fix (in PR2 follow-up patch):** added `offlineGuard()` wrapper around `mutate`/`mutateAsync` for the three online-only hooks. Checks `onlineManager.isOnline()` at click time; toast and abort if offline; nothing reaches React Query, nothing gets persisted, nothing silently replays.
   **Lesson:** any "this mutation requires connection" gate must run BEFORE `mutate()`, never inside `mutationFn`.

3. **Edit-save UI hung in "saving" state offline.** `WorkoutBuilder`'s edit-mode save initially used `syncExercises.mutate(..., { onSuccess: () => router.push(...) })` and derived `isSaving` from `mutation.isPending`. With paused mutations, `onSuccess` never fires until reconnect and `isPending` stays `true` indefinitely → save button stuck disabled, no navigation, dialog state frozen.
   **Fix (in PR2 follow-up patch):** mirrored the create-flow pattern in edit save and in three delete-dialog call sites — fire-and-forget mutate, navigate / close dialog immediately, show "Saved offline" toast at click time when offline. The optimistic update is already applied by `onMutate`, so the destination renders fresh data.
   **Lesson:** for offline-capable mutations, UI transitions (navigation, dialog close, saving spinner clear) must not wait on `onSuccess` or `isPending`. This is now codified as a project-wide rule for PR3+ to follow.

4. **Builder routes weren't pre-cached.** PR1's `WARM_ROUTES` warmed `/workouts` but not `/workouts/builder` or `/workouts/builder/[id]`. PR2's create/edit flow worked end-to-end in cache, but the user couldn't actually reach the builder offline because the SW served `/~offline` before `WorkoutBuilder` ever mounted.
   **Fix (in PR2 follow-up patch):** added `/workouts/builder` to `WARM_ROUTES` (one extra fetch per login), and extended the `syncTopDetails` warm loop to `flatMap` each top-N workout to both `/workouts/${id}` and `/workouts/builder/${id}` (10 extra fetches per login at the current N=10 cap). Same SW-control gate, same `RSC: '1'` header, same `credentials: 'same-origin'`, same fire-and-forget. No SW changes.
   **Lesson:** any new mutation flow needs its host route in `WARM_ROUTES` (or `syncTopDetails` if dynamic) — otherwise the offline-mutation work is invisible because the page can't load offline in the first place.

5. **Start-button on `/workouts` had a hard offline gate.** Pre-existing handler called `getWorkoutById` (a Server Action) at click time; the SW's `next-action` `NetworkOnly` matcher unconditionally rejects Server Actions offline, so the handler was guarded with `if (!onlineManager.isOnline()) return toast.error('Cannot start a workout while offline')`. After PR1.5 seeded `['workout', id]` for the top-N + offline-created workouts, this gate was overly aggressive.
   **Fix (in PR2 follow-up patch):** read `['workout', workoutId]` from cache first; if hit, dispatch `startWorkoutSession` directly. If miss + offline, toast "Open this workout once while online to enable offline start". If miss + online, fall back to the existing Server Action path. Applied to `WorkoutsPage` and `ClientWorkoutsTab` identically. `/workouts/[id]` already used cached data so was untouched.
   **Lesson:** dashboard pages calling Server Actions at click time are an offline foot-gun. Prefer cached reads via `queryClient.getQueryData` with a graceful fall-back when uncached.

6. **`offlineGuard()` typing.** First attempt used `unknown` for the data generic across the three online-only hooks. `useCreateWorkoutForClient`'s caller in `WorkoutBuilder` reads `data.id` from the mutate-options `onSuccess`, which broke under `unknown`. Tightened the data generic to `{ id: string } | undefined`.

- **`weeks: []` vs `plan: null` shapes.** `getAllActivePlanDashboardWeeks` returns `{ weeks: [] }` for no active plan; `getActivePlanDashboard` (per-week) returns `{ plan: null }`. PR3's optimistic deactivate/delete writes `{ plan: null }` per-key — matches the per-key shape, never the bulk shape (the bulk endpoint isn't a query key).
- **`save-all-days` deletes orphaned `PlanDayCompletion`s via cascade.** Optimistic update mirrors that — drops `weekCompletions` entries whose `planDayId` is no longer in the new payload. Without this the dashboard would show a "✓ done" badge on a day that no longer exists.
- **Server actions vs offline routes.** Existing online `createPlan`/`updatePlan`/`deletePlan`/`savePlanAllDays`/`activatePlan`/`deactivatePlan`/`skipPlanDay` server actions in `src/server/actions/plans.ts` were left unchanged — they keep working for any non-offline-aware UI path. Duplication with the new `planService` is intentional for PR3 scope; cleanup PR can delegate the actions to the service if both paths stay in lockstep.
- **Day-level `clientId` lookup precedence.** When the builder loads existing days, `LocalDay.uid = day.id` (real cuid). On save, that uid is passed as `clientId` — but the row's `PlanDay.clientId` is still NULL (never written previously), so the server's `existingDayByClientId` map doesn't match. The `dayId` precedence catches it first → update-in-place, no clientId clobber. New-day creates DO write the clientId. This pattern keeps online callers working without writing spurious clientIds to old rows.
- **`(planDayId, order)` unique constraint on `PlanDayExercise`.** Existing schema constraint. The PR3 service does delete-then-update for kept exercises with a temporary `+1000` offset on `order` to avoid mid-transaction conflicts during reorder. The original online `savePlanAllDays` always delete-recreated, so it never hit this issue.

### Blockers encountered and how they were resolved

1. **Migration could not run via `prisma migrate dev`.** Same constraint as PR2: the harness shell is non-interactive, and `migrate dev` requires a TTY for warning confirmations.
   **Fix:** hand-wrote the migration SQL file to mirror the PR2 workout-clientId migration shape, then applied via `prisma migrate deploy` (which is non-interactive by design).
   **Lesson:** the PR2 workaround is now the canonical migration pattern in this repo until the harness gains interactive-shell support.

2. **`skip-day` could deadlock on tmp dayId.** A skip-day fired against a day that was created via `save-all-days` while still offline would have a `planDayId` equal to the builder's `LocalDay.uid` (or the dashboard cache's pre-rewrite day id). `skip-day`'s `mutationFn` calls `idMap.waitFor(planDayId)` — but nothing was writing day-id mappings to `idMap`.
   **Fix:** `save-all-days` `onSuccess` now pushes every entry of `dayIdMap` AND `exerciseIdMap` into `idMap` before calling `rewritePlanDayIds`. A queued `skip-day` resolves bottom-up on reconnect: exercise-create → plan-save → skip-day.
   **Lesson:** any nested-id-returning mutation that other mutations depend on must populate `idMap` for ALL returned ids, not just the top-level one. Document this in the mutation-defaults header for future entity types.

3. **Dashboard activate had no offline render path.** Before PR3, an offline `activate` would silently update the cache only when reconnect refetched the dashboard — meaning the user's `ActivePlanSection` would show "No Active Plan" until they reconnected, even though the plan was queued to activate.
   **Fix:** added `buildSyntheticDashboard(planId)` — pulls the plan + days from the cached `['plan', id]` and synthesizes a full `ActivePlanDashboard` shape (week 1 IN_PROGRESS, no completions, `activatedAt: now`). Written into both `['activePlanDashboard', 'active']` AND `['activePlanDashboard', 1]` so the section renders immediately. Also clears every other cached `['activePlanDashboard', *]` entry in the loop to prevent stale per-week entries from a previously-active plan resurfacing during week navigation.
   **Lesson:** offline-capable mutations that drive a derived/computed view (here, the dashboard derived from plan + week + completions) need a synthesizer for the optimistic write — direct cache patches aren't enough when the source-of-truth isn't already in the cache.

4. **`existingDayById.get(r.existingId)!.exercises` ESLint warning.** The Map lookup is provably safe (we just resolved against `existingDayById.has(...)` earlier in `dayResolutions`), but `@typescript-eslint/no-non-null-assertion` flagged it.
   **Fix:** swapped to `existingDayById.get(r.existingId)?.exercises ?? []` with a comment explaining why the fallback is unreachable.
   **Lesson:** prefer `?.x ?? default` over `!` for Map lookups even when invariants make the assertion safe — keeps lint happy and makes the invariant explicit.

5. **Builder save-all-days response shape changed.** Online callers ignored everything except `totalSaved`. PR3 needed `dayIdMap` and `exerciseIdMap` on the response. Adding fields is additive but required service typing changes.
   **Fix:** new `SaveAllDaysResult` type in `services/plans.ts`. Existing online server action's return type tightens correspondingly; existing callers that destructure `result.totalSaved` continue to work without changes.

6. **`PlanBuilderPage.handleSave` already used a closure for `daysPayload`.** Adding clientIds without breaking the existing online `savePlanAllDays` server-action shape required threading `clientId?: string` (per day) and `planDayExerciseId?: string` (per exercise) through the call. Current online `savePlanAllDaysSchema` strictly enforces shape. Worked around by routing PR3 mutations exclusively through the new offline schema/route — the legacy server-action path is unreachable from the builder UI now.
